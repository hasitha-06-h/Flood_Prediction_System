About This Project

Methodology Behind the Prediction Engine

AquaGuard was built to test whether classical machine learning models, trained purely on historical meteorological readings, can offer a useful early signal for flood risk without relying on complex hydrological simulation.

Problem Statement

Traditional flood forecasting relies on hydrological models that require dense sensor networks and specialist interpretation, which many smaller districts simply don't have access to. This leaves a gap where basic weather station readings go unused for early warning.

Proposed Approach

Train and benchmark four supervised classification algorithms on a historical rainfall/temperature/humidity dataset, then deploy the strongest performer behind a lightweight web interface any field officer can use directly.

Why XGBoost

Selected for Deployment

  • Boosted trees correct each other's errors sequentially, generally giving better generalisation than a single tree or a flat ensemble average.
  • Built-in regularisation reduces overfitting risk on a dataset this small (115 rows).
  • It's a widely adopted industry standard for tabular classification tasks, making the deployed model easier to justify and maintain.

Data Preprocessing

Median imputation for missing values, IQR-based outlier capping, and standard scaling applied before any model sees the data.

Feature Set

Annual rainfall, cloud visibility, temperature, humidity and June-September seasonal rainfall - the five signals most correlated with flood outcomes in the source dataset.

Serving Layer

The trained model and scaler are serialised with Joblib and loaded once at Flask startup, keeping prediction latency well under 50ms per request.
